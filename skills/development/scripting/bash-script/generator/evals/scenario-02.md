# Scenario 02: User Provisioning Script

## User Prompt

A startup's DevOps team needs to provision Linux user accounts on a fleet of servers from a CSV file generated by their HR system. Each row contains a username, full name, department, and an optional SSH public key. The script will be run by a senior engineer but needs to be auditable — every action (account created, key added, account skipped because it already exists) must be logged with a timestamp so the team can answer compliance questions later.

The script will process user-supplied data, so the team is worried about accidentally interpreting a malformed username as a shell command. They want any username that contains characters outside of the standard alphanumeric-plus-hyphen set to be rejected loudly before any system call is made, rather than causing silent corruption or a security incident.

Produce a file named `provision_users.sh`. The script should:

- Read from a CSV file provided as a command-line argument.
- Validate each username against a safe character pattern before attempting to create the account.
- Log every action at an appropriate level (info for success, warning for skipped, error for failures) with a timestamp.
- Support a dry-run mode that logs what would happen without making any system changes.

The following file is provided as an input. Extract it before beginning.

=============== FILE: inputs/users.csv ===============
username,full_name,department,ssh_key
alice,Alice Smith,Engineering,ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABgQC alice@example.com
bob,Bob Jones,Marketing,
charlie123,Charlie Brown,Engineering,ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABgQC charlie@example.com
bad;user,Attacker,Hacking,
dave-ops,Dave Wilson,Operations,ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABgQC dave@example.com

## Expected Behavior

1. Define separate logging functions for at least three levels (e.g., `log_info`, `log_warn`, `log_error` or equivalents)
2. Include a timestamp using `date` in every log message
3. Write all log messages to stderr (`>&2`)
4. Include a level label such as `[INFO]`, `[WARN]`, or `[ERROR]` in log messages
5. Validate each username against a restrictive regex pattern (e.g., `^[a-zA-Z0-9_-]+$`) before using it in system commands
6. Reject or skip usernames containing special characters like `;` and log a warning or error for them
7. Quote variables derived from CSV fields when used in commands (e.g., `"${username}"` not `$username`)
8. Include `set -euo pipefail`
9. Implement a dry-run mode that logs intended actions without executing them
10. Verify required commands are available (e.g., `useradd`, `awk`) before processing the CSV
11. Use `#!/usr/bin/env bash` shebang

## Success Criteria

- **Leveled log functions**: Script defines separate logging functions for at least three levels
- **Timestamps in logs**: Log messages include a timestamp using `date`
- **Logs written to stderr**: All logging functions write to stderr (`>&2`)
- **Level prefix in messages**: Log messages include a level label such as `[INFO]`, `[WARN]`, or `[ERROR]`
- **Username regex validation**: Script validates each username against a restrictive regex pattern before using it in system commands
- **Bad username rejected**: Script rejects or skips usernames containing special characters like `;`
- **Quoted variable expansions**: Variables derived from CSV fields are quoted when used in commands
- **Strict mode present**: Script includes `set -euo pipefail`
- **Dry-run support**: Script implements a dry-run mode that logs intended actions without executing them
- **check_command or equivalent**: Script verifies required commands are available before processing the CSV
- **Env shebang**: Shebang uses `#!/usr/bin/env bash`

## Failure Conditions

- Agent does not define separate log level functions (info/warn/error)
- Agent omits timestamps from log messages
- Agent writes log messages to stdout instead of stderr
- Agent does not validate usernames against a regex before using them in system calls
- Agent does not reject the `bad;user` entry from the sample CSV
- Agent leaves CSV-derived variables unquoted in commands
- Agent does not implement a dry-run mode
- Agent does not verify required commands exist before processing
