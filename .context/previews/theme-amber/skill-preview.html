Overview

This skill provides deterministic TypeScript guidance for production codebases. It covers tsconfig.json tuning, advanced type constructs, and the patterns needed to maximise type safety without sacrificing developer ergonomics.

When to use this skill

  • Configuring or reviewing tsconfig.json for strict correctness
  • Writing generic utilities, mapped types, or conditional types
  • Debugging inferred vs widened type discrepancies
  • Migrating a JavaScript project to TypeScript incrementally

Compiler Configuration

Start with the strictest reasonable baseline and relax only when necessary:

{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "moduleResolution": "bundler",
    "target": "ES2022",
    "lib": ["ES2022"]
  }
}

Type Narrowing Patterns

Prefer discriminated unions over boolean flags for variant types:

type Result<T> =
  | { ok: true;  value: T }
  | { ok: false; error: string };

function handle<T>(r: Result<T>) {
  if (r.ok) {
    // r.value is T here — no cast needed
  }
}

Rule: Never use as casts in new code. If you need one, you have a modelling problem.

Utility Types Quick Reference

  • Partial<T> — all properties optional
  • Required<T> — all properties required
  • Pick<T, K> / Omit<T, K> — structural subsets
  • ReturnType<F> — extract function return type
  • Awaited<T> — unwrap Promise depth

References

tsconfig-reference

Full compiler options reference with descriptions, types, and defaults for every tsconfig.json flag.

Key flags

  • strict — enables all strict type-checking sub-flags (boolean, default false)
  • target — ECMAScript output version ("ES3"…"ESNext", default "ES3")
  • module — module system for emitted code ("CommonJS", "ESNext", "NodeNext", …)
  • moduleResolution — algorithm for resolving imports ("node", "bundler", "NodeNext")
  • outDir — redirect output structure to a directory
  • rootDir — root folder for input files; controls output layout
  • paths — path mapping entries for module aliases
  • noUncheckedIndexedAccess — adds undefined to index signature results (boolean, default false)
  • exactOptionalPropertyTypes — disallows assigning undefined to optional properties unless undefined is explicit (boolean, default false)

Recommended production baseline

{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "moduleResolution": "bundler",
    "target": "ES2022",
    "lib": ["ES2022"],
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true
  }
}

advanced-types

Conditional types, infer keyword, mapped types, template literal types, and recursive type aliases.

Conditional types

type IsArray<T> = T extends any[] ? true : false;
type Flatten<T> = T extends Array<infer Item> ? Item : T;

Mapped types

type Readonly<T> = { readonly [K in keyof T]: T[K] };
type Nullable<T> = { [K in keyof T]: T[K] | null };

Template literal types

type EventName<T extends string> = `on${Capitalize<T>}`;
// EventName<"click"> → "onClick"
type Routes = "/home" | "/about";
type ApiRoute = `/api${Routes}`; // "/api/home" | "/api/about"

Recursive type aliases

type JSONValue =
  | string | number | boolean | null
  | JSONValue[]
  | { [key: string]: JSONValue };

generics-patterns

Reusable generic patterns: constrained builders, higher-kinded type emulation, variance annotations.

Constrained generics

function pick<T, K extends keyof T>(obj: T, keys: K[]): Pick<T, K> {
  return Object.fromEntries(keys.map(k => [k, obj[k]])) as Pick<T, K>;
}

Builder pattern

class QueryBuilder<T extends object> {
  #filters: Partial<T> = {};
  where<K extends keyof T>(key: K, value: T[K]): this {
    this.#filters[key] = value;
    return this;
  }
  build(): Partial<T> { return { ...this.#filters }; }
}

Variance annotations (TS 4.7+)

type Provider<out T> = () => T;   // covariant
type Consumer<in T> = (v: T) => void; // contravariant

migration-guide

Step-by-step playbook for migrating a JavaScript codebase to TypeScript incrementally without big-bang rewrites.

Phase 1 — Enable TypeScript without breaking anything

{
  "compilerOptions": {
    "allowJs": true,
    "checkJs": false,
    "strict": false,
    "outDir": "dist"
  },
  "include": ["src"]
}

Phase 2 — Rename files incrementally

  • Rename leaf modules first (.js → .ts), working inward
  • Fix type errors one file at a time; use // @ts-ignore sparingly as escape hatches
  • Add checkJs: true for any .js files you want checked before renaming

Phase 3 — Tighten strictness

  • Enable strict: true once all files are .ts
  • Add noUncheckedIndexedAccess and fix array access patterns
  • Remove all // @ts-ignore and as any casts

Rule: Never do a big-bang rewrite. Migrate one module per PR so reviews stay tractable.

strict-mode

What each strict-mode flag does, common errors it surfaces, and recommended per-flag enable order for brownfield projects.

Flags enabled by strict: true

  • strictNullChecks — null and undefined are not assignable to other types
  • strictFunctionTypes — function parameter types checked contravariantly
  • strictBindCallApply — type-checks bind, call, apply arguments
  • strictPropertyInitialization — class properties must be initialised in constructor
  • noImplicitAny — error on expressions inferred as any
  • noImplicitThis — error when this has implicit any type
  • useUnknownInCatchVariables — catch clause variables typed unknown instead of any

Recommended brownfield enable order

  1. strictNullChecks — highest value, usually the most errors
  2. noImplicitAny
  3. strictFunctionTypes
  4. Remaining flags together (lower noise)
